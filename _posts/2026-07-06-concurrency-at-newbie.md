이 글은 신입 시절 담당했던 프로젝트에서 발생한 `수강신청 오버부킹 문제`를 해결하기 위해 시도했던 방법들을 정리한 글입니다.  
당시에는 동시성 문제에 대한 이해가 깊지 않았기 때문에 `synchronized`, `BlockingQueue`, `SELECT FOR UPDATE` 등을 차례로 적용하며 문제를 해결하려고 했습니다.  

돌이켜보면 멀티 컨테이너 환경에서 JVM 내부 동시성 제어 도구를 사용하는 데 명확한 한계가 있었고, 실제 정합성은 DB Lock과 비즈니스 로직 보완을 통해 개선되었습니다.  
따라서 이 글은 특정 방법이 정답이라는 내용보다는, 운영 환경에서 동시성 문제를 마주하며 어떤 시행착오를 겪었고 무엇을 배웠는지 정리하는 회고에 가깝습니다.

---
## 동시성 문제 상황

수강신청이 시작되면 평소보다 훨씬 많은 사용자가 동시에 접속했고, 특정 강좌에서 정원을 초과하여 승인되는 문제가 발생했습니다.  
특히 교차신청 기능이 추가된 이후에는 기존 강좌 취소, 신규 강좌 신청, 후보자 자동 승인 로직이 함께 실행되면서 특정 상황에서 정원보다 많은 인원이 승인되는 문제가 발생했습니다.

예를 들어 정원이 20명인 강좌에 대해 여러 사용자가 거의 동시에 신청하면, 각 요청은 조회 시점에는 정원이 남아 있다고 판단할 수 있습니다.  
하지만 실제 등록 처리 시점에는 다른 요청이 먼저 승인되면서, 여러 요청이 모두 승인되어 정원을 초과하는 상황이 발생할 수 있습니다.

결국 문제의 핵심은 조회 시점의 강좌 상태와 등록 처리 시점의 강좌 상태가 달라지는 `Race Condition`이었습니다.



---
## 프로젝트 구조
수강신청 동시성 문제가 발생한 프로젝트는 WEB 서버 2대, WAS 서버 2대로 구성되어 있었고, 각 WAS 서버에서는 Spring Boot 애플리케이션 컨테이너가 2개씩 실행되고 있었습니다.  
즉, 총 4개의 Spring Boot 인스턴스가 요청을 처리하는 구조였고, DB는 Oracle을 사용했습니다.

각 컨테이너는 별도의 JVM 프로세스로 실행되기 때문에 메모리를 공유하지 않습니다.  
따라서 Java 애플리케이션 내부의 `synchronized`, `BlockingQueue`, `Map`과 같은 객체는 컨테이너별로 독립적으로 생성됩니다.

## 문제해결 시도 1 - Synchronized
첫 번째로 시도한 방법은 수강신청 메소드 전체를 `synchronized`로 묶는 것이었습니다.  
단일 JVM 내부라면 `synchronized`를 통해 동시에 하나의 스레드만 임계 영역에 진입하도록 제한할 수 있습니다.

하지만 이 프로젝트는 총 4개의 Spring Boot 컨테이너가 실행되는 구조였습니다.  
`synchronized`는 JVM 내부에서만 동작하므로, 컨테이너가 여러 개인 환경에서는 각 컨테이너마다 별도의 Lock이 생성됩니다.

결과적으로 특정 컨테이너 내부에서는 하나의 요청만 실행되더라도, 다른 컨테이너에서는 동일한 수강신청 로직이 동시에 실행될 수 있었습니다.  
따라서 `synchronized`만으로는 전체 서버 기준의 동시성 문제를 해결할 수 없었습니다.


---
## 문제해결 시도 2 - Blocking Queue
두 번째로 시도한 방법은 `BlockingQueue`를 이용해 요청을 순서대로 처리해보는 것이었습니다.  
컨트롤러에서 요청을 받으면 수강신청에 필요한 파라미터를 Queue에 넣고, 서비스에서는 Queue에서 값을 꺼내 처리하는 방식으로 접근했습니다.

처음에는 Queue를 사용하면 요청이 순서대로 처리될 것이라고 생각했습니다.  
하지만 지금 돌이켜보면 `BlockingQueue` 역시 JVM 내부 메모리에 생성되는 객체입니다.  
따라서 컨테이너가 4개인 환경에서는 Queue 역시 4개가 생성됩니다.

```text
Container-1 → BlockingQueue-1
Container-2 → BlockingQueue-2
Container-3 → BlockingQueue-3
Container-4 → BlockingQueue-4
```

즉, `BlockingQueue`만으로는 전체 요청 순서나 선착순 처리를 보장할 수 없었습니다.  
각 컨테이너 내부에서는 Queue 순서가 유지될 수 있지만, 전체 시스템 관점에서는 요청이 여러 Queue로 분산되기 때문입니다.

이 시도는 결과적으로 멀티 컨테이너 환경에서 JVM 내부 Queue만으로는 전체 동시성 제어를 할 수 없다는 한계를 확인한 과정이었습니다.

## 문제해결 시도 3 - 비관적 락(SELECT FOR UPDATE)
세 번째로 시도한 방법은 DB의 `SELECT FOR UPDATE`를 이용한 비관적 락이었습니다.  
애플리케이션 내부 Lock이나 Queue는 컨테이너 간에 공유되지 않지만, DB Lock은 모든 컨테이너가 동일하게 바라보는 DB 자원에 대해 동작합니다.

처음에는 별도의 Lock 용도 테이블을 만들고, 수강신청 로직이 시작될 때 해당 테이블에 `SELECT FOR UPDATE`를 걸어 전체 로직을 직렬화하려고 했습니다.

```sql
SELECT 1
FROM LOCK_TABLE
FOR UPDATE;
```

이 방식은 동일한 DB Row에 대해 하나의 트랜잭션만 Lock을 획득할 수 있기 때문에, 여러 컨테이너에서 동시에 요청이 들어와도 하나씩 처리되도록 만들 수 있었습니다.

하지만 문제가 있었습니다.  
수강신청 요청이 많아지면 Lock을 얻지 못한 요청들이 DB Connection을 점유한 채 대기하게 되었고, 그 결과 Connection Pool 고갈이나 DB 접속 오류가 발생할 수 있었습니다.

또한 모든 수강신청 요청을 하나의 Lock으로 묶으면 서로 다른 강좌에 대한 신청까지 모두 대기하게 됩니다.  
즉, 정합성은 어느 정도 확보할 수 있었지만, 병목이 너무 큰 방식이었습니다.

---
## 문제해결 시도 4 - BlockingQueue + SELECT FOR UPDATE Row Lock
세 번째 시도에서 가장 큰 문제는 모든 요청이 하나의 Lock을 바라본다는 점이었습니다.  
그래서 Lock 범위를 전체 테이블 수준이 아니라 강좌 단위 Row 수준으로 줄이는 방향으로 변경했습니다.

Lock 테이블을 강좌번호 기준으로 구성하고, 수강신청 대상 강좌에 해당하는 Row에만 `SELECT FOR UPDATE`를 수행했습니다.

```sql
SELECT 1
FROM LECTURE_LOCK
WHERE lec_seq = :lecSeq
FOR UPDATE;
```

이렇게 하면 동일한 강좌에 대한 신청은 하나씩 처리되지만, 서로 다른 강좌에 대한 신청은 동시에 처리될 수 있습니다.

```text
강좌 A 신청 → 강좌 A Row Lock
강좌 B 신청 → 강좌 B Row Lock
강좌 C 신청 → 강좌 C Row Lock
```

이 방식에서 `BlockingQueue`는 전체 요청 순서를 보장하기 위한 장치가 아니었습니다.  
컨테이너별로 생성되는 Queue이기 때문에 전체 서버 기준의 단일 대기열 역할을 할 수 없기 때문입니다.

대신 `BlockingQueue`는 각 컨테이너 내부에서 DB 접근 요청이 순간적으로 몰리는 것을 완화하기 위한 보조 장치로 사용했습니다.  
실제 동일 강좌에 대한 정합성은 `SELECT FOR UPDATE`를 이용한 DB Row Lock이 담당했습니다.

## 문제해결 시도 5 - 수강신청 로직 보완
동시성 제어 방법을 적용하는 것과 별개로, 수강신청 비즈니스 로직 자체도 다시 확인했습니다.

그 과정에서 교차신청과 후보자 자동 승인 로직이 함께 실행될 때 일부 케이스가 정확히 처리되지 않는 문제가 있었습니다.  
예를 들어 후보 상태의 사용자가 다른 강좌로 이동하는 경우에도 기존 강좌의 후보자 1명을 승인자로 올리는 로직이 실행될 수 있었습니다.

이 로직을 수정한 이후에는 별도의 동시성 제어를 제거하고 테스트해도 오버부킹 현상이 크게 줄었습니다.  
결과적으로 동시성 문제를 해결할 때 Lock이나 Queue를 적용하기 전에, 비즈니스 로직 자체가 정확한지 먼저 확인하는 것이 중요하다는 점을 배웠습니다.

## 추가로 고려한 방법 - 추첨형 수강신청
앞선 시도들을 통해 오버부킹 문제는 어느 정도 완화할 수 있었지만, 서버 스펙과 DB Connection 수를 고려했을 때 사용자가 더 증가하면 다시 문제가 발생할 수 있는 잠재적 위험이 있었습니다.

그래서 기술적인 해결과 별개로, 수강신청 정책 자체를 변경하는 방법도 제안했습니다.  
기존 방식은 사용자가 신청하는 즉시 승인 여부를 확인하는 선착순 방식에 가까웠습니다.  
이 구조에서는 특정 시점에 요청이 몰릴 수밖에 없고, 서버와 DB에 순간적인 부하가 집중됩니다.

대안으로 제안한 방식은 `추첨형 수강신청`이었습니다.  
특정 기간 동안 신청을 받은 뒤, 관리자가 신청자를 추첨하여 결과를 문자나 이메일로 안내하는 방식입니다.

이 방식은 순간적인 트래픽 집중을 줄일 수 있고, 고객이 원했던 “더 다양한 사용자가 수강 기회를 얻는 구조”와도 맞았습니다.  
다만 실제 적용은 고객과 회사 사정으로 진행되지 않았습니다.

## 후기
이번 경험을 통해 가장 크게 배운 점은 멀티 컨테이너 환경에서는 JVM 내부 동시성 도구만으로 전체 정합성을 보장할 수 없다는 점입니다.

`Synchronized`, `BlockingQueue`, `Map`은 모두 애플리케이션 내부 메모리에 존재합니다.  
따라서 컨테이너가 여러 개라면 각 컨테이너마다 별도로 생성되고, 전체 서버 기준의 단일 Lock이나 단일 Queue 역할을 할 수 없습니다.

반면 DB Lock은 모든 컨테이너가 공유하는 DB 자원을 기준으로 동작하기 때문에, 동일 강좌에 대한 동시성 제어에는 더 적합했습니다.  
하지만 Lock 범위를 너무 크게 잡으면 Connection Pool 고갈이나 병목 문제가 발생할 수 있기 때문에, 가능한 한 작은 단위로 Lock 범위를 줄이는 것이 중요했습니다.

결과적으로 이 경험은 이후 Redis 기반 대기열, 분산락, 선착순 쿠폰 발급 같은 동시성 주제를 이해하는 데 큰 도움이 되었습니다.  
지금 다시 설계한다면 애플리케이션 내부 Queue만으로 처리하기보다는, Redis Queue나 메시지 브로커처럼 여러 인스턴스가 공유할 수 있는 외부 대기열 구조를 우선 검토할 것 같습니다.

무엇보다 동시성 문제를 해결할 때는 Lock을 적용하기 전에 비즈니스 로직 자체가 정확한지 먼저 확인해야 한다는 점을 배웠습니다.  
실제로 이번 사례에서도 로직 보완이 오버부킹을 줄이는 데 큰 영향을 주었습니다.
